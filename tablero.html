Estados de Historias de Usuario – Enfoque Célula

Resumen operativo para control real del avance, alineado a Azure DevOps y al trabajo diario de la célula.

Resumen de Estados (visión célula)

  • Nueva / Aprobada: La célula no puede trabajar. Pendiente de negocio.
  • Refinada: La célula entiende, acepta y se compromete.
  • Desarrollo: DEV trabajando activamente.
  • Fin Desarrollo: DEV terminó y entrega a QA.
  • Pruebas QA: QA valida. DEV solo soporta.
  • Certificado QA: Lista para negocio.
  • Pruebas / Certificado UAT: Validación y aprobación del negocio.
  • Producción: Despliegue y estabilización.
  • Cerrada: HU estable, capacidad liberada.

Estados Especiales

  • Bloqueada: Impedimento real. El Scrum Master debe gestionar desbloqueo.
  • Cancelada: Ya no aplica. Se libera capacidad inmediatamente.

Regla de Oro

Si la célula no puede avanzar, el estado está mal.

Los estados reflejan realidad operativa, no intención ni presión.

Es de vital importancia la actualización de tablero con las notas en la sección de @notas, es responsabilidad de cada miembro del equipo velar por la actualización de estados, Bloqueos, Preguntas y aclaraciones.

Ningun comentario sobra todas las opiniones son tenidas en cuenta, La ausencia de notas si puede llevar a un problema.

Gestión de BUGs – Azure DevOps

Gestión de BUGs – Azure DevOps

Modelo operativo de estados, responsables y caminos

Los BUGs se gestionan como Work Items independientes. El flujo prioriza claridad de responsabilidad, trazabilidad y control real del estado.

Flujo Oficial de Estados

Nueva

QA / Usuario

BUG registrado y pendiente de gestión inicial.

Activa

Scrum Master DEV / LT / Configuración

• El Scrum Master mueve de Nueva → Activa para continuar el flujo y puede asignar al Analista de Configuración o al DEV según el caso.

• El DEV / LT y/o Analista de Configuración devuelve el BUG a Activa cuando requiere notas aclaratorias o información faltante.

Nota obligatoria y accionable.

Proveedor

DEV / LT Analista Configuración

Estado de corrección.
DEV, LT o Analista de Configuración trabajan activamente el BUG.

El Analista de Configuración evalúa:
• Si es tema de proveedor → permanece o pasa a Proveedor
• Si es tema de ambiente/configuración → pasa a Resuelta

Resuelta

DEV / Configuración

El BUG pasa a Resuelta cuando:
• El DEV envía correo de liberación
• Se confirma solución por ambiente/configuración

Cerrada

QA

QA valida y confirma la corrección.

Reabierto

QA

La corrección no cumple. El BUG retorna a Proveedor.

Cancelada

Scrum Master

BUG no aplica, duplicado o inválido.
